CSSE Final Reflection

By Yuna Lee — Class of 2025

Learning Through Teamwork

At the beginning of this CSSE class, I honestly felt a little lost. Everyone seemed to know what they were doing — GitHub, terminal commands, writing methods — while I was still trying to figure out how to run the program at all. But what made this class special for me was the people I worked with. I didn’t just "learn" programming. I learned how to lean on my team.

I had moments where I was stuck, completely unsure what to do next. That’s where my group stepped in. Whether it was helping me figure out a broken branch or explaining what a pull request actually does, they were patient and explained things in ways I could understand. Eventually, I stopped being afraid to ask for help — and started finding confidence in giving help back.

GitHub: A platform for hosting and collaborating on code.

What We Built (and Broke and Fixed)

Our group worked together to plan, build, and test different projects. We started using GitHub to track versions of our code, and I learned how to use branches to safely try new ideas without messing up the main project. I didn’t understand how merging worked at first — and yes, I broke the main branch once. But I learned how to fix it, how to test before I push, and how to write better commit messages. We also learned about deployment: seeing our projects actually live online felt real.

Branches: versions of a project that allow for isolated development.
Merging: Combining code from one branch into another.
Deployment: Putting your finished project online so people can use it.

Growing from Reviews

At first, I was nervous about code reviews — what if someone pointed out something I completely missed? But over time, I saw how useful those conversations were. I got better at receiving feedback and also giving it. Our retrospectives helped us reflect on what went well and what needed to change. Presenting live demos also pushed me out of my comfort zone and helped me grow in confidence, not just as a coder, but as a communicator.

Code reviews: Class checking your code for errors, style, and improvements.
Retrospectives: Talking with your team about what went well and what could be better.

Understanding the Foundations

We covered essential data types like numbers, strings, booleans, arrays, and JSON objects. These might sound basic, but they’re the building blocks of everything we did. I learned how to use operators to combine strings or perform calculations, and how to use boolean logic in control structures.

Before this class, I didn’t really get how loops or `if` statements worked. But writing games and apps using control structures like `for` loops and `if/else` helped it finally click.

Numbers: Numeric values like integers or decimals.
Strings: Text inside quotes.
Booleans: Values that are either true or false.
Arrays: Ordered lists of elements.
JSON objects: Data in key-value pair format.
Operators: Special symbols that let the computer do things like math or comparisons. (e.g., +, -, &&).
Control structures: Tools like if/else and loops that control code flow.

Input, Output, and Interaction

One of the coolest parts was learning how to get input from users using HTML forms, and then using JavaScript to validate that input or trigger things with key events. Working with the DOM made everything feel interactive — like building something real, not just typing into a console.

HTML Forms: A way to gather data from people using text fields, checkboxes, and more.
Key events: When the computer reacts to someone pressing a key on the keyboard.
DOM (Document Object Model): The part of the page your code interacts with to make changes happen live.

Object-Oriented Concepts

Classes were confusing to me at first. Writing a class, adding methods, instantiating objects — it felt like too much. But after enough repetition and support from my group, I finally understood how to use them. We created objects with different behaviors and properties, and I saw how powerful it is when you structure your code cleanly.

Classes: Structured templates used to define the properties and behaviors of objects.
Methods: Functions defined within a class that describe what an object can do.
Objects: Individual instances created from a class, each with its own data and behavior.
Instantiation: The process of creating an object based on a class definition.

Practices That Stuck

One of the biggest takeaways from this year was the Single Responsibility Principle (SRP) — making sure each class or function only has one job. We also learned about Object Literals and how they differ from Instances. I especially enjoyed learning how to build FSMs (Finite State Machines) for games. Using inheritance was something I struggled with but started to grasp as the year went on.

SRP (Single Responsibility Principle): Each piece of code should have one job only.
Object Literal: An object created directly using curly braces { }.
Instance: A specific object created from a class.
FSM (Finite State Machine): A system with limited states (like start, play, end).
Inheritance: When one class takes on properties of another.
